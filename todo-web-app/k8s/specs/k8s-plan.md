## Summary

Phase 4 containerizes the existing TodoAI full-stack application (FastAPI backend + Next.js 16 frontend) using Docker multi-stage builds and orchestrates them on a local Minikube cluster via a single Helm chart. The deployment is designed to satisfy SC-002 (≤3 terminal commands from prerequisites to running application) while enforcing secret isolation, resource constraints, and health probe gating throughout. `NEXT_PUBLIC_API_URL` is baked into the frontend image at build time using the backend's Minikube NodePort URL, satisfying both browser-side and SSR access patterns without requiring a running reverse proxy or `kubectl port-forward` process.

## Docker Build Strategy

### Backend: Multi-Stage python:3.13-slim

**Builder Stage** (`builder`):
- Base: `python:3.13-slim`
- Install `uv` via the official binary copy pattern: `COPY --from=ghcr.io/astral-sh/uv:latest /uv /usr/local/bin/uv`
- Set `WORKDIR /app`
- Copy `pyproject.toml` and `uv.lock` first (layer cache optimization)
- Run `uv sync --frozen --no-dev` — installs all production dependencies into `/app/.venv`
- Copy remaining application source (`app/`)

**Runner Stage** (`runner`):
- Base: `python:3.13-slim`
- Create non-root user: `RUN adduser --system --no-create-home --uid 1001 appuser`
- Set `WORKDIR /app`
- Copy `.venv` from builder: `COPY --from=builder /app/.venv /app/.venv`
- Copy application source: `COPY --from=builder /app/app /app/app`
- Set `ENV PATH="/app/.venv/bin:$PATH"` so uvicorn is on PATH
- `USER 1001` (non-root)
- `EXPOSE 8000`
- `CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000", "--workers", "1"]`

**Backend `.dockerignore`**:
```
.venv/
__pycache__/
*.pyc
*.pyo
.env
.env.*
tests/
.git/
.gitignore
migrations/
*.egg-info/
dist/
build/
.pytest_cache/
.mypy_cache/
```

### Frontend: Multi-Stage node:20-alpine

**Deps Stage** (`deps`):
- Base: `node:20-alpine`
- Set `WORKDIR /app`
- Copy `package.json` and `package-lock.json`
- Run `npm ci --omit=dev` — installs production dependencies only

**Builder Stage** (`builder`):
- Base: `node:20-alpine`
- Set `WORKDIR /app`
- Copy `node_modules` from `deps` stage
- Copy all remaining source files (next.config.mjs, tsconfig.json, src/, etc.)
- Declare `ARG NEXT_PUBLIC_API_URL` — this is the critical build-time variable
- Set `ENV NEXT_PUBLIC_API_URL=$NEXT_PUBLIC_API_URL`
- Run `npm run build` — inlines NEXT_PUBLIC_API_URL into the bundle during compilation

**Runner Stage** (`runner`):
- Base: `node:20-alpine`
- Create non-root user: `RUN addgroup --system --gid 1001 nodejs && adduser --system --uid 1001 nextjs`
- Set `WORKDIR /app`
- Copy `.next/standalone` from builder
- Copy `.next/static` from builder to `.next/static`
- Copy `public/` from builder if it exists
- `USER nextjs`
- `EXPOSE 3000`
- `ENV PORT=3000`
- `ENV NODE_ENV=production`
- `CMD ["node", "server.js"]` (standalone output entry point)

**Note on Next.js standalone output**: `next.config.mjs` must include `output: 'standalone'` for the runner stage to work with `node server.js`.

**Frontend `.dockerignore`**:
```
node_modules/
.next/
.env
.env.*
.env.local
.git/
.gitignore
README.md
*.log
.playwright-mcp/
```

## Deployment Commands (SC-002: ≤3 Commands)

**Command 1** — Build images into Minikube's Docker daemon:
```bash
eval $(minikube docker-env) && docker build -t todo-backend:local ./todo-web-app/backend/ && docker build --build-arg NEXT_PUBLIC_API_URL=http://$(minikube ip):30800 -t todo-frontend:local ./todo-web-app/frontend/
```

**Command 2** — Deploy the full stack:
```bash
helm upgrade --install todoai ./todo-web-app/k8s/charts/todoai --values ./todo-web-app/k8s/charts/todoai/secrets.values.yaml
```

**Command 3** — Open the application in a browser:
```bash
minikube service todoai-frontend
```

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|--------------------------------------|
| New `todo-web-app/k8s/` directory outside the approved `backend/` and `frontend/` layout (Principle VII — Monorepo) | Phase 4 introduces Helm chart infrastructure, Kubernetes manifests, and K8s-specific documentation that are categorically different from application source code. Co-locating them in `backend/` or `frontend/` would mix infrastructure concerns with application concerns, violating separation of concerns and making the chart harder to find and maintain. | Placing Helm charts inside `frontend/` (e.g., `frontend/k8s/`) was considered and rejected because the chart manages both services; it has no stronger affinity to the frontend than the backend. Creating a top-level `k8s/` at repo root was considered and rejected because the chart is specific to the `todo-web-app` monorepo sub-project and should be colocated with its subject. `todo-web-app/k8s/` is the minimum viable path that keeps infrastructure code with its subject without mixing layers. |

## Follow-ups and Risks

- **Risk 1 — `next.config.mjs` standalone output**: Must add `output: 'standalone'` before building the frontend image. Absence causes runtime failure (`node server.js` not found) even though the build succeeds.
- **Risk 2 — Minikube NodePort reachability from WSL2**: `$(minikube ip)` may not be reachable from the Windows browser on some WSL2 + Docker driver configurations. Use `minikube service todoai-frontend --url` or `minikube tunnel` as fallback.
- **Risk 3 — CORS_ORIGINS secret formatting**: Must exactly match the browser URL including port. Incorrect value causes all API calls to fail with CORS errors — verify with browser DevTools Network tab after deployment.
