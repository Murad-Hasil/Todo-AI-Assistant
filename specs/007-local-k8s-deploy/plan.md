# Implementation Plan: Local Kubernetes Deployment (Phase 4)

**Feature Branch**: `007-local-k8s-deploy` | **Date**: 2026-03-05 | **Spec**: [specs/007-local-k8s-deploy/spec.md](./spec.md)
**Input**: Feature specification from `/specs/007-local-k8s-deploy/spec.md`

---

## Summary

Phase 4 containerizes the existing TodoAI full-stack application (FastAPI backend + Next.js 16 frontend) using Docker multi-stage builds and orchestrates them on a local Minikube cluster via a single Helm chart. The deployment is designed to satisfy SC-002 (≤3 terminal commands from prerequisites to running application) while enforcing secret isolation, resource constraints, and health probe gating throughout. `NEXT_PUBLIC_API_URL` is baked into the frontend image at build time using the backend's Minikube NodePort URL, satisfying both browser-side and SSR access patterns without requiring a running reverse proxy or `kubectl port-forward` process.

## Docker Build Strategy

### Backend: Multi-Stage python:3.13-slim

**Builder Stage** (`builder`):
- Base: `python:3.13-slim`
- Install `uv` via the official binary copy pattern: `COPY --from=ghcr.io/astral-sh/uv:latest /uv /usr/local/bin/uv`
- Set `WORKDIR /app`
- Copy `pyproject.toml` and `uv.lock` first (layer cache optimization)
- Run `uv sync --frozen --no-dev` — installs all production dependencies into `/app/.venv`
- Copy remaining application source (`app/`)

**Runner Stage** (`runner`):
- Base: `python:3.13-slim`
- Create non-root user: `RUN adduser --system --no-create-home --uid 1001 appuser`
- Set `WORKDIR /app`
- Copy `.venv` from builder: `COPY --from=builder /app/.venv /app/.venv`
- Copy application source: `COPY --from=builder /app/app /app/app`
- Set `ENV PATH="/app/.venv/bin:$PATH"` so uvicorn is on PATH
- `USER 1001` (non-root)
- `EXPOSE 8000`
- `CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000", "--workers", "1"]`

**Backend `.dockerignore`**:
```
.venv/
__pycache__/
*.pyc
*.pyo
.env
.env.*
tests/
.git/
.gitignore
migrations/
*.egg-info/
dist/
build/
.pytest_cache/
.mypy_cache/
```

### Frontend: Multi-Stage node:20-alpine

**Deps Stage** (`deps`):
- Base: `node:20-alpine`
- Set `WORKDIR /app`
- Copy `package.json` and `package-lock.json`
- Run `npm ci --omit=dev` — installs production dependencies only

**Builder Stage** (`builder`):
- Base: `node:20-alpine`
- Set `WORKDIR /app`
- Copy `node_modules` from `deps` stage
- Copy all remaining source files (next.config.mjs, tsconfig.json, src/, etc.)
- Declare `ARG NEXT_PUBLIC_API_URL` — this is the critical build-time variable
- Set `ENV NEXT_PUBLIC_API_URL=$NEXT_PUBLIC_API_URL`
- Run `npm run build` — inlines NEXT_PUBLIC_API_URL into the bundle during compilation

**Runner Stage** (`runner`):
- Base: `node:20-alpine`
- Create non-root user: `RUN addgroup --system --gid 1001 nodejs && adduser --system --uid 1001 nextjs`
- Set `WORKDIR /app`
- Copy `.next/standalone` from builder (Next.js output mode: standalone must be configured in `next.config.mjs`)
- Copy `.next/static` from builder to `.next/static`
- Copy `public/` from builder if it exists
- `USER nextjs`
- `EXPOSE 3000`
- `ENV PORT=3000`
- `ENV NODE_ENV=production`
- `CMD ["node", "server.js"]` (standalone output entry point)

**Note on Next.js standalone output**: `next.config.mjs` must include `output: 'standalone'` for the runner stage to work with `node server.js`. This is an additive change to the Next.js config that does not affect the development or test workflow.

**Frontend `.dockerignore`**:
```

.next/
.env
.env.*
.env.local
.git/
.gitignore
README.md
*.log
.playwright-mcp/
```

### CORS Configuration

The `CORS_ORIGINS` environment variable (injected from `todoai-backend-secret`) must include the frontend's external URL. The value to use is:

```
http://$(minikube ip):30300
```

Where `$(minikube ip)` is evaluated at Helm install time by the developer and substituted into `secrets.values.yaml`. The backend FastAPI application reads `CORS_ORIGINS` from pydantic-settings and uses it to configure the `CORSMiddleware` — no application code changes required.

### DNS Resolution Summary

| Access Pattern | URL | Resolution Method |
|----------------|-----|-------------------|
| Browser → Backend | `http://$(minikube ip):30800` | Minikube NodePort on host network |
| Browser → Frontend | `http://$(minikube ip):30300` | Minikube NodePort on host network |
| Frontend SSR → Backend (via baked URL) | `http://$(minikube ip):30800` | Same NodePort (pod has host network access via Minikube) |
| Internal pod-to-pod (direct) | `http://todoai-backend-svc:8000` | Kubernetes CoreDNS ClusterIP |

## Image Build and Load Workflow

The following sequence satisfies SC-002 (≤3 terminal commands after prerequisites are met). Commands 1 and 2 are combined into a single compound command.

**Step 1 — Point Docker daemon at Minikube and build both images:**
```bash
eval $(minikube docker-env) && \
  docker build \
    -t todo-backend:local \
    ./todo-web-app/backend/ && \
  docker build \
    --build-arg NEXT_PUBLIC_API_URL=http://$(minikube ip):30800 \
    -t todo-frontend:local \
    ./todo-web-app/frontend/
```

**Step 2 — Deploy with Helm:**
```bash
helm upgrade --install todoai \
  ./todo-web-app/k8s/charts/todoai \
  --values ./todo-web-app/k8s/charts/todoai/secrets.values.yaml
```

**Step 3 — Open frontend in browser:**
```bash
minikube service todoai-frontend
```

**Notes**:
- `eval $(minikube docker-env)` configures the current shell's Docker client to communicate with Minikube's Docker daemon. Images built in this shell context are available inside Minikube without a registry push.
- `imagePullPolicy: Never` in both Deployments ensures Kubernetes never attempts to pull these local images from an external registry.
- The `uv.lock` file must be present in `todo-web-app/backend/` at build time — it is committed to the repository.
- If `next.config.mjs` does not already include `output: 'standalone'`, this must be added before the frontend Dockerfile build will work with the `node server.js` runner pattern.

### Cluster Inspection Strategy

1. After `helm install`, run `kubectl-ai "Are all pods healthy?"` to verify rollout
2. If pods are not Ready, run `kubectl-ai "Why is the [pod-name] pod not ready?"` for AI-generated diagnosis
3. For runtime errors, run `kubectl-ai "Show the last 50 log lines for the todoai-backend pod"` 
4. For memory pressure, run `kubectl-ai "Which pods are consuming the most memory?"` and compare against limits

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|--------------------------------------|
| New `todo-web-app/k8s/` directory outside the approved `backend/` and `frontend/` layout (Principle VII — Monorepo) | Phase 4 introduces Helm chart infrastructure, Kubernetes manifests, and K8s-specific documentation that are categorically different from application source code. Co-locating them in `backend/` or `frontend/` would mix infrastructure concerns with application concerns, violating separation of concerns and making the chart harder to find and maintain. | Placing Helm charts inside `frontend/` (e.g., `frontend/k8s/`) was considered and rejected because the chart manages both services; it has no stronger affinity to the frontend than the backend. Creating a top-level `k8s/` at repo root was considered and rejected because the chart is specific to the `todo-web-app` monorepo sub-project and should be colocated with its subject. `todo-web-app/k8s/` is the minimum viable path that keeps infrastructure code with its subject without mixing layers. |

## Follow-ups and Risks

- **Risk 1 — `next.config.mjs` standalone output**: The frontend Dockerfile's runner stage depends on `output: 'standalone'` in Next.js config. If this key is absent, the build completes but `node server.js` fails at runtime with a module not found error. This must be confirmed and added in Step 1 of the implementation sequence before any image build is attempted.
- **Risk 2 — Minikube NodePort reachability from WSL2**: On some WSL2 configurations, `$(minikube ip)` resolves to a Minikube-internal address not directly reachable from the Windows browser. In this case, `minikube service todoai-frontend --url` may return a different URL via `minikube tunnel`-style port forwarding. The quickstart must document both paths and provide a `minikube tunnel` fallback.
- **Risk 3 — CORS_ORIGINS secret formatting**: The `CORS_ORIGINS` value is a comma-separated string that must exactly match the URL the browser uses to load the frontend (including port number). If the developer copies the example value without substituting the real `$(minikube ip)`, CORS will reject all frontend API calls. The quickstart must include an explicit CORS configuration verification step.
